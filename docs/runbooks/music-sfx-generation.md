# Music and SFX generation

Quick-start commands for generating background music loops and generic gameplay SFX
locally. Background, quality bar, and how generated clips get wired into the game live in
`docs/CONTENT-GENERATION.md`. NPC dialogue TTS (a different tool) lives in
`docs/runbooks/audio-generation.md`.

This runbook covers tooling at `~/aop-ai-tools/`, which is **not part of this repo** — it
runs on the operator's machine and writes output directly into
`apps/web/public/audio/music/` and `apps/web/public/audio/sfx/`.

## Two different tools for two different asset types

- **Music** (menu/exploration/battle loops): generated with **MusicGen**
  (`facebook/musicgen-small`, via `transformers`), run locally on MPS/CPU. A generative
  model earns its keep here — melody, instrumentation, and mood are exactly what it's
  good at, and there's no cheap procedural substitute for "sounds like a song."
- **SFX** (UI click, combat hit, ship movement, coin pickup, notification chime):
  generated by **procedural synthesis** (numpy/scipy — sine tones, filtered noise,
  envelopes), not a model. Short (80ms–1s) UI/impact blips are cheap to synthesize
  directly and a generative audio model is both overkill and harder to control precisely
  (exact pitch, exact envelope shape) than writing the waveform math. Don't reach for
  MusicGen/AudioLDM for this category — see "Why not a model for SFX?" below.

## Prerequisites

- Python env already set up at `~/aop-ai-tools/venv/` (same venv as Piper TTS; the
  `torch`/`torchaudio`/`transformers`/`scipy`/`numpy` versions already installed there are
  sufficient — no separate venv was needed).
- This machine is Apple Silicon (MPS, no CUDA). `torch.backends.mps.is_available()` is
  checked at runtime and used automatically; falls back to CPU otherwise.
- First run of the music script downloads `facebook/musicgen-small` from Hugging Face
  (~1.5GB) — needs network access once, then it's cached under `~/.cache/huggingface/`.

## Quick start — music

```bash
source ~/aop-ai-tools/venv/bin/activate
python ~/aop-ai-tools/generate_game_music.py
```

Generates all 3 tracks (menu/exploration/battle) in a single batched `model.generate()`
call — batching gave a large wall-clock win on this hardware (see "Performance notes"
below) over generating them one at a time. Output lands in:

```
apps/web/public/audio/music/menu_theme.wav
apps/web/public/audio/music/exploration_ambient.wav
apps/web/public/audio/music/battle_theme.wav
```

To change mood/instrumentation, edit the `TRACKS` prompt list at the top of the script.
To change track length, edit `DURATION_SEC` (28s by default — see performance notes before
going much higher).

### How looping works

MusicGen has no concept of "seamless loop" — raw output doesn't connect cleanly back to
its own start. The script generates `DURATION_SEC + CROSSFADE_SEC` of audio, then applies
`loop_crossfade()`: a linear self-crossfade that blends the tail into the head. The result
is sample-continuous at both junctions of the loop (proven by construction, not by ear —
see the function's docstring), so `audio.loop = true` in the client plays it back with no
audible seam regardless of what the raw generation sounded like at its true start/end.

## Quick start — SFX

```bash
source ~/aop-ai-tools/venv/bin/activate
python ~/aop-ai-tools/generate_game_sfx.py
```

Regenerates all 5 clips instantly (no model, pure signal processing — this runs in well
under a second). Output lands in:

```
apps/web/public/audio/sfx/ui_click.wav
apps/web/public/audio/sfx/combat_hit.wav
apps/web/public/audio/sfx/ship_movement.wav
apps/web/public/audio/sfx/coin_pickup.wav
apps/web/public/audio/sfx/notification_chime.wav
```

To tweak a sound, edit its `gen_*()` function directly (frequency, envelope decay,
duration) — no need to touch anything else, each clip is an independent function.

## Curate before committing

```bash
afplay apps/web/public/audio/music/menu_theme.wav
afplay apps/web/public/audio/sfx/coin_pickup.wav
```

Discard and regenerate (don't commit) anything that fails the quality bar in
`docs/CONTENT-GENERATION.md` — dead air, wrong mood/tempo for the scene, or (for music) an
audible seam at the loop point despite the crossfade (can happen if the raw generation's
last `CROSSFADE_SEC` seconds are wildly different in character from the first — try a
different prompt/seed, or increase `CROSSFADE_SEC` for a longer blend). MusicGen's
`do_sample=True` means every run is a different take; there's no seed pinning in the
script today, so "regenerate" means rerunning the whole batch (cheap — a few minutes).

## Performance notes (Apple Silicon, no CUDA)

Measured on this machine (M5, 16GB RAM, MPS backend):

- Model load: ~5-35s (first load per process; cached weights after the first run).
- Single-track generation: ~12.8s of wall clock per second of audio generated
  (`musicgen-small`, `guidance_scale=3.0`).
- **Batching all 3 music prompts into one `model.generate()` call is dramatically
  faster than generating them sequentially** — token-by-token decoding parallelizes
  across the batch dimension, so 3 tracks batched together take roughly the same wall
  clock as 1 track alone. All 3 28-second tracks batched together took ~8 minutes total.
  If you add more tracks, keep them batched rather than looping `generate()` calls.
- Longer clips cost more per-token as the sequence grows (attention/KV-cache overhead),
  so time doesn't scale perfectly linearly with duration — a 90-second loop would take
  meaningfully longer than 3x a 30-second one. 25-35s was the sweet spot found here
  (a few minutes per batch); going much past that risks crossing into "tens of minutes,"
  which isn't worth it for a loop that repeats anyway.
- SFX generation (procedural) is effectively instantaneous — no model, no GPU/MPS
  involvement at all.

## Why not a model for SFX?

AudioLDM/AudioLDM2/Stable Audio Open were considered for the 5 short SFX clips (UI click,
combat hit, ship movement, coin pickup, notification chime). Rejected in favor of
procedural synthesis because:

- These are all short (80ms-1s), simple, high-pitch-precision sounds — exactly what
  sine/noise/envelope synthesis does natively and a diffusion-style audio model doesn't
  control precisely (you'd prompt-engineer and re-roll for a specific pitch/timing rather
  than just writing `sine_tone(0.09, 1568)`).
- Zero generation time and zero curation-roulette — the output is deterministic and
  tunable by editing a number, not by re-prompting and hoping.
- No second heavyweight model/dependency to install for assets that don't benefit from
  one. `numpy`/`scipy` were already present in the venv from the MusicGen setup.

If a future SFX need doesn't fit clean synthesis (e.g. a complex environmental ambience
bed, a crowd/cheer sound), reconsider a text-to-audio model at that point — this is a
per-asset-type judgment call, not a blanket rule.

## Known gaps

- No seed pinning in `generate_game_music.py` — every regeneration is a fresh take.
  Add a `torch.manual_seed()` call if you need to reproduce a specific take exactly.
- `TOKEN_BUFFER` in `generate_game_music.py` exists because measured tokens/sec ran
  slightly under the nominal 50Hz EnCodec frame rate; if you see the "shrinking loop
  target" warning printed at generation time, the buffer wasn't enough for that
  duration — bump `TOKEN_BUFFER` up.
